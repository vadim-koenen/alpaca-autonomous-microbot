# P2-011B — Coinbase Fill Response Discovery Report

**ADVISORY / READ-ONLY / LOCAL SOURCE ONLY**
Generated by `scripts/coinbase_fill_response_discovery.py`

This report answers the seven gating questions for P2-011B before any hook is written.
All conclusions are derived from static inspection of the current tree (HEAD 1c81139 + P2-011A).

## 1. Which Coinbase response currently contains actual fill/proceeds/fee facts?
**broker_coinbase.py:get_order_status (via self._client.get_order)**

The `self._client.get_order(order_id=...)` response (normalised inside `get_order_status`) is the **only** place in the entire codebase that surfaces:
- `filled_size`
- `average_filled_price`
- `filled_value`
- `total_fees`
- `completion_percentage`
- `settled`
- `last_fill_time`

`place_limit_order` and `place_market_order` return only `{order_id, success_response}` with no fill data (order is still pending).

## 2. Are average_filled_price, filled_size, and fee fields direct Coinbase facts or local estimates?
**Direct facts** when read from `broker_coinbase.py:get_order_status`.

**Local estimates** everywhere else:
- `position_manager.py:_execute_exit` computes `fees_paid` exclusively from `get_cfg("crypto", "fee_entry_pct")` + `fee_exit_pct` (currently 0.6% in Coinbase config).
- `journal.py:log_exit` receives and persists only those pre-computed estimates.
- Entry `total_fees` is captured into `session.open_positions[...]['total_fees']` but is **not** written to the durable journal as an immutable fact row.

## 3. Can sell proceeds be captured directly from current broker responses?
**No.**

There is no code path that calls `get_order_status` on an exit/close order_id after `close_position()` succeeds.
In `_execute_exit`, `exit_price` is the decision-time market price, not the actual fill price of the market sell.
After `log_exit`, the position is immediately removed from tracking. No post-close reconciliation exists for sell legs.
All prior reconciliation reports (P2-007 etc.) correctly report zero direct sell proceeds in journal EXIT rows.

## 4. Are partial fills possible, and should the future logger write one row per order or one row per fill?
**Yes — order object carries completion_percentage + cumulative filled_size/average_filled_price. Current paths only act on terminal 'filled'.**

Coinbase Advanced Trade order objects support partial fills via `completion_percentage` + cumulative fields.
For the bot's $1 exploration trades on BTC/ETH/SOL this is uncommon but possible.

**Recommendation for logger:** One row per order (when it reaches terminal 'filled' status), capturing the final cumulative values + `raw_order_response_json`. A dedicated `/fills` list endpoint would be required for true per-fill rows; none is called anywhere today.

## 5. Could repeated get_order_status polling duplicate rows?
**HIGH — _reconcile_pending_orders and _backfill_missing_order_status poll repeatedly while order_status not in _TERMINAL_ORDER_STATUSES. After 'filled' the order_id remains queryable until position is dropped.**

Both `_reconcile_pending_orders` (every loop) and `_backfill_missing_order_status` (startup) call `get_order_status` while the order is not yet terminal.
Once 'filled', the order_id remains in the position dict until the position is dropped after exit.
A future hook **must** implement strong deduplication (order_id + terminal status, or a small persisted set of already-logged order_ids).

## 6. Which exact function is the safest future hook?

**Recommended seam:**
> position_manager.py:_reconcile_pending_orders (and _backfill_missing_order_status) immediately after `status_info = get_order_status(order_id)` when normalized == 'filled'. This is the only place where direct Coinbase facts are turned into trusted session state for entries. journal.py:log_exit receives only pre-computed estimates. No post-close get_order_status call exists for exit legs.

### Why not the plan candidates?
- `broker_coinbase.py:get_order_status` — too low-level; would require passing logger into the broker, would fire on every poll + dry_run/paper/Alpaca paths.
- `broker_coinbase.py:place_*_order` — no fill facts present at placement time.
- `journal.py:log_exit` — receives only estimates; no actual exit-leg fill data.

**Gap for complete coverage:** No equivalent post-close capture exists for sell orders. Wiring only entries would still leave realized P/L incomplete.

## 7. What tests are required before a future hook patch is allowed?
- Mock broker.get_order_status returning 'filled' + partial states; assert logger called exactly once per unique order_id across repeated polls and simulated restarts (via saved positions).
- Test that no rows are appended for dry_run, paper, or when broker lacks get_order_status.
- Test deterministic capture of raw_order_response_json (full dict from get_order).
- Test deduplication guard (order_id + 'filled' status or processed set).
- Verify exit path still has zero direct sell proceeds/fill facts (negative test).
- Add to existing position_manager reassociation tests.

## Overall Assessment

The local codebase **does prove** that direct fill facts exist — but only for **entry** orders, and only inside the position reconciliation logic.
Direct sell proceeds and actual exit fees are **not** available from current broker response handling.

Therefore the correct next step is **not** to wire the logger to live execution yet.
P2-011C (or a tightly scoped follow-up) should first add a narrow, read-only capture of the raw `get_order` payload at the identified seam, plus a fixture-based test that proves we can reconstruct gross/net P/L from the captured facts.

This keeps the safety invariant: no hook is installed until we have proven, with local evidence, that the required facts are actually present for both legs.
