## 8. Recommended Immutable Idempotency Key
**Preferred:** `account_mode + product_id + order_id + (trade_id or entry_id)`
From fixtures: `entry_id` or `trade_id` in fills list provides stable per-fill identity.
`order_id` alone is insufficient for orders with multiple fills.

## 9. Recommended Future Append-Only Fill Logging Path
**Recommended combination:**
- Primary: historical fills REST path (for per-fill price, size, fee, liquidity, stable IDs)
- Secondary: order/status path (for final cumulative confirmation and exit order capture)
- Reconciliation layer: match fills to their parent order
- Future: WebSocket user stream for near-real-time (lower latency)
Current order/status path alone is **insufficient** for high-confidence per-fill P/L and sell proceeds.

## 10. Logger Hook Safety Verdict
**BLOCKED**

Reasons from fixtures and code inspection:
- Direct per-fill fees and liquidity role only appear in historical fills payload (not currently wrapped or called).
- Direct sell proceeds for exit legs are reconstructible from order-level payload *if* the exit order status is polled (current code does not do this for close_position fills).
- Stable fill-level idempotency key (trade_id/entry_id) requires the fills list endpoint.
- Payloads missing fees or stable IDs exist and must be handled gracefully.

## Summary Recommendation
The safest future path is to add a minimal `get_historical_fills(...)` wrapper in BrokerCoinbase,
then implement a narrow capture point (likely in position_manager after exit reconciliation)
that uses both order status + fills list, with reconciliation.
Keep the logger hook blocked until the above is proven with real captured payloads and tests.
